Testes E2E de PR — do label ao vídeo, sem mãos humanas

Um label na PR dispara um teste ponta a ponta num device Android real. Ninguém aperta play, ninguém instala nada na mão, ninguém assiste ao vivo. O resultado — vídeo, screenshots e veredito — volta sozinho para dentro da PR.

Parte 1 · O fluxo hoje

Seis passos, zero intervenção

O gatilho é o label e2e:run numa PR. A partir daí a esteira roda sozinha até devolver evidência dentro da própria conversa da PR.

O pulso percorre os seis estágios em loop; ao chegar no comentário, a evidência é montada e o ciclo recomeça. Sem prefers-reduced-motion, é o único elemento animado por conta própria.

Parte 2 · Onde está o valor

Para produto e para engenharia

O mesmo teste serve aos dois times por motivos diferentes: engenharia confia porque o teste prova a mudança, não só passa; produto confia porque vê o comportamento sem precisar rodar nada.

O teste prova o fix, não a si mesmo

O mesmo cenário roda contra o app sem o fix e contra o app com o fix. Se as duas pistas derem o mesmo resultado, o teste não discrimina nada — só existe registro, não prova.

Prova diferencial · mesmo teste, duas versões do app
sem o fix

✗ falhou
com o fix
voltou à home
✓ passou
Se o teste passa nos dois, ele não testa nada.

A flag vem do próprio teste

O cenário declara a feature flag que precisa, por um canal de override que só existe em builds de teste. Sem sorteio de rollout, sem tocar o painel de produção. 

- launchApp:
    arguments:
      rcOverrides:
        nova_home_v2: true

Evidência que qualquer pessoa entende

Vídeo do device e screenshots aparecem direto no comentário da PR. Um reviewer de produto vê o comportamento sem instalar nada, sem pedir print, sem esperar ninguém narrar.

Experimental nunca bloqueia

A lane nova é informativa. Uma falha de infra — device offline, timeout de instalação — não vira PR travada; ela aparece como aviso, não como bloqueio.

Parte 4 · Entregue: a spec nasce no ClickUp

A spec de teste agora vem do card, não da PR

A ingestão direta está ligada em produção: o control-plane resolve o código AC-xxxx a partir do título da PR, busca a descrição do card correspondente no ClickUp e usa a spec de lá. O corpo da PR continua valendo como fallback de primeira classe — não uma solução provisória, uma rota permanente para quando o card não tem spec. Cada job grava de onde a spec veio: acceptance_criteria_source: clickup ou pr_body.

PR → AC-xxxx → card ClickUp → spec test-intent/v1
Prova real: a PR piloto #1074 rodou verde com acceptance_criteria_source: clickup — a spec saiu 100% do card, sem nenhum aviso.

Um formato canônico: test-intent/v1

A spec vive dentro de um bloco de código cercado cuja primeira linha é exatamente spec: test-intent/v1. O card pode ter prosa ao redor — o pipeline extrai só o bloco. Formato mergeado em konsi-mobile-test-platform (PR #20), documentado em docs/test-intent-spec.md. Cada job grava também o formato reconhecido: acceptance_criteria_format: test-intent/v1 ou freeform.

Template · test-intent/v1
spec: test-intent/v1
clickup_id: AC-XXXX
user_story: >-
  Como <persona>, quando <contexto>, quero <ação/resultado>.
env:
  lane: pre_login
  flags:
    FF_nome_da_flag:
      value: true
      mechanism: rcOverrides
criteria:
  - id: AC-XXXX-C1
    kind: e2e
    given: <estado inicial do app>
    when: <passos do usuário>
    then: <resultado observável na UI>
    oracle: ui_state
    risk: read_only
    semantics_ids: [<identifiers dos elementos críticos>]
  - id: AC-XXXX-C2
    kind: unit
    given: <condição>
    when: <variação>
    then: <comportamento esperado>
    covered_by: <onde a cobertura unit mora>

Obrigatório: user_story e ao menos um critério em criteria, cada um com id no padrão AC-<n>-C<n>, kind (e2e ou unit), given, when e then — sem isso o gate reprova. Opcional recomendado: env.lane, env.flags, oracle, risk, semantics_ids, constraints, covered_by.

Gate obrigatório no CI

Com a label e2e:run, o check Gate de spec E2E reprova a PR quando não existe um bloco canônico válido nem no card, nem no corpo da PR. Entrou pela PR #1080 no konsi-app, que segue aberta: recebeu 1 aprovação em 18/jul e aguarda a segunda revisão para poder mergear. Ainda não está em produção.

Parte 5 · Fan-out multi-teste por PR

Uma PR, vários testes, ao mesmo tempo

Até 21/jul, uma PR com N casos de teste rodava todos em sequência, dentro de um único job, num único device — o segundo caso só começava quando o primeiro terminasse. Desde 22/jul, o control-plane divide o trabalho: cria 1 job pai — que nunca executa teste, só coordena — e N jobs filho, um por caso de teste, que podem rodar em devices diferentes ao mesmo tempo. A PR continua recebendo um único Check agregado no final, como sempre.

Fan-out · 1 job pai → N jobs filho → 1 Check agregado
job pai coordenador nunca executa teste — só orquestra
↓↓↓
filho 1 na fila device A
filho 2 na fila device B
filho 3 na fila aguarda vaga liberar
Check agregado da PR: aguardando…

O cap de execuções simultâneas por PR é configurável (MAESTRO_MAX_ACTIVE_CHILDREN_PER_PARENT, padrão 2) — por isso o terceiro filho espera uma vaga liberar antes de começar. O diagrama roda em loop; sem prefers-reduced-motion, ele para no estado final (os três passaram, Check verde).

No ar desde 22/jul: plataforma PR #29 (KonsiLabs/konsi-mobile-test-platform), mergeada e em produção — deploy confirmado às 15:39 UTC do dia 22/jul (run 29934291511). 329 testes automatizados cobrem o fan-out; zero mudança no agente que roda dentro do device — todo o trabalho novo é no control-plane. A entrega também corrigiu um bug real de produção: um campo None era persistido como a string "null", o que deixava um finished_at fantasma em jobs reabertos.

O que ainda falta — por isso a manchete é "em produção", não "validado com devices": existe uma PR de validação pronta, konsi-app #1084, com exatamente 3 casos de teste test-intent/v1 (abrir Ajuda, abrir Ajustes, rolar o dashboard até o topo). O esperado é 2 rodando em paralelo e 1 na fila — mas essa prova física ainda não rodou: o host da farm no escritório caiu em 22/jul, e os 2 devices da lane post_login ficaram desconectados do USB. Assim que o host voltar e a label e2e:run for aplicada na #1084, essa prova acontece.

O que já dá confiança de que o paralelismo funciona de verdade, fisicamente: em 21/jul (PR #24) o mesmo teste rodou ao mesmo tempo em 2 devices físicos diferentes — um moto g35 e um Galaxy A15, mesmo APK — e as duas execuções deram passou, em 37s e 44s. E o host está preparado desde 20/jul (Maestro 2.7.0, uma porta de driver por worker) para paralelismo real de execução, não só de agendamento.

A semana na farm (20–22/jul)

O fan-out foi o topo de uma sequência de entregas na mesma semana — cada uma resolvendo um problema concreto de operar uma farm de devices de verdade.

EntregaPRDataO que mudou
Alocação paralela de devices #22 20/jul O sistema aprendeu a reservar vários celulares ao mesmo tempo com leases (empréstimos com dono e prazo), um scheduler que prioriza PR sobre trabalho de fundo e um worker por aparelho físico. +4.127 linhas em 31 arquivos — é a fundação sobre a qual o fan-out foi construído.
Menos alarme falso #24, #25 21/jul A farm passou a distinguir "o app tem bug" de "o ambiente não estava pronto". Se o APK não tem o canal de feature flag esperado, ou se o celular "esqueceu" a sessão logada, o teste vira "pulado por pré-condição" (Check neutro) em vez de reprovar a PR. Ressalva: os 2 devices da lane post_login seguem aguardando religamento e reaquecimento manual.
Celulares nunca ficam parados #26 21/jul Quando a fila de PR está vazia, os devices rodam testes leves de manutenção, só leitura (filler). Cada execução também passou a registrar uma "impressão digital" das feature flags ativas, para auditoria.
Máquina de build dedicada — 22/jul O build do APK para E2E ganhou um runner de CI próprio (ubuntu-24.04-4core-e2e, 4 cores/16GB) no konsi-app. Provisionado, estreia junto com a validação física — nenhum build real ainda o usou.
Parte 6 · Próximos passos

Roadmap curto, sem datas

Compilador determinístico intent → YAML

A IA gera a intenção estruturada; quem escreve o YAML final é código determinístico. O fan-out já cobriu uma fatia disso: o novo app/slicing.py deriva os testes de forma determinística a partir do reference_flow de cada spec, com validações de catálogo antes de virar job. Não é peça 100% entregue — falta estender esse determinismo para o restante da geração de YAML —, mas é uma base real que o fan-out já validou em produção.

Glossário
lane pre_login
Trilha de testes que roda antes do fluxo de login, cobrindo as telas iniciais do app.
Prova diferencial
Rodar o mesmo teste contra a versão com e sem o fix. Só conta como prova se falha numa e passa na outra.
validated-only
Modo em que só entram no relatório oficial os resultados de execuções já confirmadas como confiáveis.
Test Intent
A intenção estruturada de um teste — o quê e por quê — separada do YAML final gerado a partir dela.
test-intent/v1
Formato canônico da spec: bloco de código cercado cuja primeira linha é exatamente "spec: test-intent/v1". O pipeline extrai só esse bloco, mesmo com prosa em volta no card.
feature-map
Mapeamento entre áreas do código/diff e os testes E2E que cobrem cada área, usado para selecionar o que rodar.
Semantics
Identificadores estáveis de elementos na tela, usados pelo teste para localizar o que precisa tocar ou ler.
job pai / job filho
No fan-out, o job pai coordena e nunca executa teste; cada job filho roda exatamente um caso de teste, podendo usar um device diferente dos irmãos.
fan-out
Dividir uma PR com N casos de teste em N execuções (jobs filho) que podem rodar em paralelo, em vez de uma única execução sequencial.
cap de concorrência
Limite de jobs filho ativos ao mesmo tempo por PR. Configurável via MAESTRO_MAX_ACTIVE_CHILDREN_PER_PARENT, padrão 2.
filler
Teste leve de manutenção, só leitura, que ocupa um device quando a fila de PR está vazia — para o aparelho nunca ficar parado.
lease de device
Reserva de um device físico com dono e prazo, usada pelo scheduler para alocar vários celulares ao mesmo tempo sem dois jobs colidirem no mesmo aparelho.
